# Phase 03.1: Code Review Report

**Reviewed:** 2026-04-23
**Depth:** standard
**Files Reviewed:** 10
**Status:** issues_found

## Summary

This phase implements a multi-signal defense module (`rag/defense.py`), extends the generator with a DEF-02 system prompt (`rag/generator.py`), wires defenses into the pipeline (`rag/pipeline.py`), and adds an evaluation + ablation harness (`scripts/run_eval.py`, `scripts/_build_ablation_table.py`, `scripts/train_defense.py`).

The implementation is largely solid. The two critical issues both concern the evaluation harness: an unguarded `KeyError` on `metadata["passage_id"]` that will silently crash the eval loop and corrupt results for any query whose chunk lacks that key, and a training-time contamination risk where the Signal 4 retrieval-z feature is set to a uniform constant for all train/val examples (masking the signal entirely during LR fitting). Both could skew reported ASR/defense numbers.

The warnings cover logic correctness in the defense filtering threshold comparison, a bare `except:` in the perplexity loop that swallows all errors including OOM, a double-count risk in `_imperative_ratio`, the train/test split relying on an unshuffled `all_clean` list, and the ablation table script using an unsafe `open()` without a context manager. The info items are code quality notes.

## Critical Issues

### CR-01: Unguarded `metadata["passage_id"]` crashes eval loop on missing key

**File:** `scripts/run_eval.py:171-183`

**Issue:** The tier-detection block accesses `h["metadata"]["passage_id"]` via direct dict subscript on every hit. If any retrieved chunk has a `metadata` dict that lacks the `passage_id` key (e.g., chunks inserted by older indexing code, or chunks from a re-indexed collection where the key was serialized differently by ChromaDB), this raises a `KeyError` mid-loop, the exception propagates out of `pipeline.query()`, and the entire evaluation run aborts — or worse, the exception is caught silently further up the stack and that query's result is missing from `results`, silently deflating `n` and skewing every reported rate.

The `rag/retriever.py` metadata comment documents the key as `{"passage_id": int, "chunk_idx": int}` but ChromaDB stores metadata as string-valued JSON; if the value was stored as a string `"passage_id"` key that round-trips fine, but there is no defensive `.get()` anywhere in this path.

**Fix:**
```python
# Replace the four tier_retrieved assignments (lines 170-185) with:
def _pid(hit: dict) -> int:
    return hit.get("metadata", {}).get("passage_id", -1)

tier1_retrieved = any(TIER1_ID_START <= _pid(h) < TIER2_ID_START for h in hits)
tier2_retrieved = any(TIER2_ID_START <= _pid(h) < TIER3_ID_START for h in hits)
tier3_retrieved = any(TIER3_ID_START <= _pid(h) < TIER4_ID_START for h in hits)
tier4_retrieved = any(_pid(h) >= TIER4_ID_START for h in hits)
```

### CR-02: Signal 4 retrieval-z is a constant during LR training — signal is effectively ablated from the meta-classifier

**File:** `scripts/train_defense.py:540-581`

**Issue:** In `main()`, all train and val examples are assigned `retrieval_score = 0.7` (line 543). After Signal 4 calibration, `default_retrieval_z = (0.7 - mu) / std` is computed (line 578) and then **every row in `X_train[:, 3]` and `X_val[:, 3]` is overwritten with this single constant** (lines 579-580). A feature column with zero variance cannot be used by `StandardScaler` + `LogisticRegression` to learn a useful weight — the scaler will produce a zero-divided or near-zero scaled column, and the LR coefficient for feature 3 will be arbitrary (driven by regularization to near-zero). The resulting `FusedDefense` meta-classifier is therefore a 3-signal classifier in practice, not 4-signal, even though it is presented as fused 4-signal in all ablation results.

This does not affect the `fingerprint` single-signal ablation (which applies Signal 4 directly with its own threshold), but it undermines the `fused` row's claim to be a genuine 4-signal ensemble.

**Fix:**

Assign per-example retrieval scores that reflect the true distribution. For training positives, use the score value stored in `all_positives` items (currently hardcoded to `0.9`); for negatives, use `0.7`. The dummy baseline is fine for the first LR fit, but after real calibration re-extract features with real scores rather than overwriting column 3 with a constant:

```python
# After calibrate_signal4(), recompute X_train and X_val with real retrieval scores
# by passing the actual per-example scores (already available in train_examples lists).
# Instead of:
#   X_train[:, 3] = default_retrieval_z
# do:
train_val_scores = [0.9 if label == 1 else 0.7 for _, label in train_examples + val_examples]
mu = signal4_baseline["mu"]
std = max(signal4_baseline["std"], 1e-9)
for i, rs in enumerate(train_val_scores):
    X_trainval[i, 3] = (rs - mu) / std
X_train = X_trainval[:n_train]
X_val   = X_trainval[n_train:]
```

## Warnings

### WR-01: `_imperative_ratio` over-counts — trailing empty sentence after final punctuation

**File:** `rag/defense.py:114-119`

**Issue:** `re.split(r"[.!?]+", text)` always produces an empty-string trailing element when the text ends with sentence-ending punctuation (e.g., `"Paris is France."` splits to `["Paris is France", ""]`). The empty string is counted in `total` (inflating the denominator) and can never match an imperative verb. This biases `imperative_ratio` slightly downward for all text that ends with punctuation, which is the common case for natural sentences but not for injection payloads (which often end without punctuation or with unusual punctuation). The directional effect makes the signal marginally less sensitive to clean text and slightly more sensitive to injections — a benign bias — but the calculation is incorrect and inconsistent with the stated semantics of "fraction of sentences containing imperative verbs."

The same bug is duplicated in `scripts/train_defense.py:300` (`_imperative_ratio_score`).

**Fix:**
```python
sentences = [s.strip() for s in re.split(r"[.!?]+", text) if s.strip()]
total = max(len(sentences), 1)
```

### WR-02: Bare `except Exception` in perplexity loop silently swallows OOM and other fatal errors

**File:** `rag/defense.py:76-94` and `scripts/train_defense.py:272-288`

**Issue:** The `except Exception: return 0.0` in `_compute_perplexity_max_nll` catches *all* exceptions including `torch.cuda.OutOfMemoryError`, `RuntimeError: CUDA error`, and `MemoryError`. When the model is out of memory or the GPU fails, the function silently returns `0.0` — the same value as an empty text. This means a failing defense signal is indistinguishable from a legitimately low-perplexity chunk, so injections may pass the filter without any visible error. During training in `scripts/train_defense.py`, this produces silently wrong feature rows.

The docstring says this is "DoS mitigation (T-03.1-04)" but the catch is too broad.

**Fix:** Catch only the anticipated inference errors, and log or re-raise unexpected ones:
```python
except (RuntimeError, ValueError) as exc:
    # Truncation/inference errors — treat as 0.0 (DoS mitigation T-03.1-04)
    return 0.0
# Let MemoryError and SystemExit propagate
```

### WR-03: `SingleSignalDefense.filter` uses `<=` threshold comparison — zero-score chunks pass even when they should not

**File:** `rag/defense.py:513`

**Issue:** The filter keeps chunks where `score <= threshold`. For the `imperative` signal, the default threshold is `0.1`, so a chunk with `imperative_ratio == 0.1` (exactly one imperative sentence in a 10-sentence chunk) is **kept** rather than removed. The intended semantics from the docstring ("Any imperative sentence = flagged") is `< threshold` or `threshold = 0.0`. The off-by-one on the boundary means chunks with exactly `1/N` imperative sentences slip through for any `N` where `1/N == 0.1`.

For `fingerprint`, threshold `2.0` with `<=` means a z-score of exactly 2.0 is kept; this is also likely wrong (targeted poison should score above 2.0 and be removed).

Compare with `FusedDefense._is_injection` which uses strict `>` (line 340).

**Fix:** Change to strict less-than to be consistent with `FusedDefense`:
```python
return [c for c in chunks if self._score_single(c) < effective_threshold]
```
Or explicitly document that the boundary is inclusive-keep and adjust the threshold constants accordingly.

### WR-04: `load_training_data` splits `all_clean` without seeding the shuffle — train/test split is non-reproducible between Python versions

**File:** `scripts/train_defense.py:131`

**Issue:** `random.shuffle(all_clean)` is called at line 131. While `main()` calls `random.seed(42)` before invoking `load_training_data()`, the order in which `all_clean` is populated depends on the iteration order of JSONL records, which may differ between corpus versions or Python dict/set ordering changes. More importantly, if `load_training_data()` is called in isolation (e.g., in a notebook or a test), the seed is not set, producing a different split each time. Different splits mean different test-clean samples (lines 133: `test_clean = all_clean[300:450]`), which can cause test set contamination or variation in reported F1/FPR numbers.

**Fix:** Pass the seed explicitly and use it locally:
```python
def load_training_data(seed: int = 42):
    ...
    rng = random.Random(seed)
    rng.shuffle(all_clean)
```

### WR-05: `_build_ablation_table.py` writes output file without a context manager — file may be unclosed on exception

**File:** `scripts/_build_ablation_table.py:74`

**Issue:** `json.dump(table, open(OUTPUT, "w"), indent=2)` uses a bare `open()` without a `with` statement. If `json.dump` raises (e.g., serialization error on a NaN float in the table), the file handle is leaked and the file may be partially written without being flushed/closed. On Windows this also blocks subsequent reads of the file until GC collects the handle.

**Fix:**
```python
with open(OUTPUT, "w") as f:
    json.dump(table, f, indent=2)
```

### IN-03: `_verify_defense.py` expects `FusedDefense(models_dir="models")` to raise `FileNotFoundError` — will silently pass if models exist

**File:** `scripts/_verify_defense.py:35-40`

**Issue:** The acceptance check on lines 35-40 expects that `FusedDefense(models_dir="models")` raises `FileNotFoundError` because "no trained model yet." Once models are trained and `models/bert_classifier/config.json` exists, this check will fail (no exception raised), causing the verification script to print `"ERROR: should have raised FileNotFoundError"` and exit 1. The script is now effectively broken as a smoke test for the trained state.

**Fix:** Either remove this stale check, or gate it with a runtime check: `if not Path("models/bert_classifier/config.json").exists(): ...`.
