## Summary

Phase 2.1 builds a small, fully-local, reproducible RAG pipeline that every downstream phase (attacks, defense, evaluation) depends on. The stack is already locked in CLAUDE.md and CONTEXT.md: `sentence-transformers` (all-MiniLM-L6-v2) + `ChromaDB` + `Ollama` (llama3.2:3b) with no LangChain/LlamaIndex abstraction. The research focus is therefore *how* to wire these three libraries together correctly, avoid the well-known pitfalls (wrong distance metric, tokenizer mismatch, non-deterministic runs, huge NQ download), and produce a plan that fits inside the ~24 hours of calendar time remaining before the Phase 2 submission deadline on 2026-04-12.

Three findings drive the plan more than anything else. **First, the full `google-research-datasets/natural_questions` dataset is 144.87 GB on disk** — unusable on a laptop. The project must use the pre-processed `sentence-transformers/natural-questions` variant (44 MB, `{query, answer}` pairs where `answer` is the Wikipedia passage), which is perfect for seeding 500 clean passages into ChromaDB. **Second, ChromaDB defaults to L2 distance, not cosine** — every tutorial that ignores `configuration={"hnsw": {"space": "cosine"}}` silently produces wrong rankings for normalized text embeddings. **Third, Python 3.11 is not currently installed on this machine** (only 3.13 is) — the plan must include a conda environment creation step, because CLAUDE.md pins to 3.11 and several downstream libraries have rough edges on 3.13.

**Primary recommendation:** Create a conda env `rag-security` with Python 3.11, install the pinned stack (torch 2.11, sentence-transformers 5.4, chromadb 1.5.7, ollama 0.6.1, datasets 4.8.4, transformers 5.5.3), load 500 passages from `sentence-transformers/natural-questions` (dedup on `answer`), chunk each with a word-based 256/32 splitter, embed with MiniLM (`normalize_embeddings=True`), index into a ChromaDB `PersistentClient` at `./.chroma/` using `configuration={"hnsw": {"space": "cosine"}}`, generate with Ollama `chat()` (already running, `llama3.2:3b` already pulled), and log every retrieval as a single JSONL line via an append-only writer. Use a TOML config file (`config.toml`) — Python 3.11 has `tomllib` in stdlib, so no extra dependency.

### Pattern 1: Explicit embedding with ChromaDB `PersistentClient`

**What:** Encode with sentence-transformers, pass embeddings explicitly to ChromaDB, use cosine distance.

**When to use:** Always for this project. Do NOT let ChromaDB call its own embedding function — we need full control for the attack phase (we must be able to embed adversarial chunks with the exact same model).

**Example:**
```python
# Source: https://docs.trychroma.com/docs/collections/configure (cosine metric)
#         sentence-transformers docs for encode() API
import chromadb
from sentence_transformers import SentenceTransformer

encoder = SentenceTransformer("sentence-transformers/all-MiniLM-L6-v2")
client = chromadb.PersistentClient(path="./.chroma")

collection = client.get_or_create_collection(
    name="nq_clean_v1",
    configuration={"hnsw": {"space": "cosine"}},  # CRITICAL: default is l2
    metadata={"description": "NQ 500 passages, 256-token chunks"},
)

# Batch-encode with normalization (cosine-friendly)
embs = encoder.encode(chunks, batch_size=32, normalize_embeddings=True,
                      show_progress_bar=True, convert_to_numpy=True)

collection.add(
    embeddings=embs.tolist(),
    documents=chunks,
    ids=[f"chunk_{i:05d}" for i in range(len(chunks))],
    metadatas=[{"passage_id": p, "chunk_idx": c} for p, c in chunk_meta],
)
```

### Pattern 2: Idempotent indexing (skip if already built)

**What:** Check collection size before re-indexing. Jupyter users will re-run cells frequently — re-embedding 500 passages each time wastes 30 s and produces duplicate IDs.

**When to use:** Any "load corpus" cell in the notebook.

**Example:**
```python
def ensure_indexed(client, encoder, passages, collection_name="nq_clean_v1"):
    coll = client.get_or_create_collection(
        name=collection_name,
        configuration={"hnsw": {"space": "cosine"}},
    )
    expected_chunks = sum(1 for _ in chunk_all(passages))
    if coll.count() == expected_chunks:
        print(f"Collection already has {coll.count()} chunks — skipping index.")
        return coll
    if coll.count() > 0:
        # Stale state — drop and rebuild for reproducibility
        client.delete_collection(collection_name)
        coll = client.create_collection(
            name=collection_name,
            configuration={"hnsw": {"space": "cosine"}},
        )
    # ... embed and add ...
    return coll
```

### Pattern 3: Retrieval logger (append-only JSONL)

**What:** Each `query()` call writes exactly one line to `logs/retrieval.jsonl`. Append mode survives notebook re-runs and process restarts.

**When to use:** Inside `Retriever.query()` (or wherever the retrieval step actually happens in `pipeline.py`).

**Log schema (one JSON per line):**
```json
{
  "timestamp": "2026-04-11T14:23:05.123456+00:00",
  "query": "what is the boiling point of water",
  "top_k": 5,
  "collection": "nq_clean_v1",
  "results": [
    {"rank": 1, "chunk_id": "chunk_00042", "score": 0.8431, "text": "Water boils at 100°C at standard atmospheric pressure...", "metadata": {"passage_id": 17, "chunk_idx": 0}},
    {"rank": 2, "chunk_id": "chunk_00113", "score": 0.7892, "text": "...", "metadata": {"passage_id": 42, "chunk_idx": 1}}
  ]
}
```

**`score` field:** ChromaDB returns `distances` (lower = more similar for cosine, in `[0, 2]` range). Convert to similarity `score = 1 - distance` and log that — it's more intuitive and matches published attack papers. Document the conversion in `rag/logger.py` docstring.

### Pattern 4: Ollama chat with RAG context

**What:** Use `ollama.chat()` with a system prompt that frames retrieved chunks as untrusted context. This is a *baseline* prompt — Phase 3.1's defense may rewrite it.

**Example:**
```python
from ollama import Client, ResponseError
client = Client(host="http://localhost:11434")

system_prompt = (
    "You are a helpful assistant. Answer the user's question using ONLY "
    "the information in the CONTEXT section below. If the context does "
    "not contain the answer, say so."
)
context_block = "\n\n".join(f"[{i+1}] {c}" for i, c in enumerate(retrieved_chunks))
user_prompt = f"CONTEXT:\n{context_block}\n\nQUESTION: {query}"

try:
    resp = client.chat(
        model="llama3.2:3b",
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": user_prompt},
        ],
        options={"temperature": 0.0, "seed": 42},  # determinism
    )
    answer = resp["message"]["content"]
except ResponseError as e:
    if e.status_code == 404:
        raise RuntimeError(
            "Model llama3.2:3b not found. Run: ollama pull llama3.2:3b"
        ) from e
    raise
```

**Key details:**
- Use `Client(host=...)` (not the module-level `ollama.chat()`) so tests can point at a mock.
- Pass `temperature=0.0` AND `seed=42` for reproducibility.
- Catch `ResponseError` status 404 = model missing; give a clear install command.
- Do NOT use `format="json"` — llama3.2:3b is flaky at JSON mode and the attack phase needs free-form output.

### Pattern 5: Seeded reproducibility

**What:** Single `set_seed(seed)` function called once at pipeline init and once in tests.

**Example:**
```python
# rag/pipeline.py
import os, random
import numpy as np
import torch

def set_global_seed(seed: int = 42) -> None:
    os.environ["PYTHONHASHSEED"] = str(seed)
    random.seed(seed)
    np.random.seed(seed)
    torch.manual_seed(seed)
    torch.cuda.manual_seed_all(seed)
    # HF datasets honors NUMPY seed for .shuffle(seed=...) — pass explicitly instead
```

**Gotcha:** `datasets.Dataset.shuffle()` takes its own `seed` parameter — the global numpy seed does NOT propagate. Always pass `seed=config.seed` to shuffle calls explicitly.

**Gotcha:** sentence-transformers encoder is deterministic on CPU with same input + same torch seed, but NOT across torch versions or CPU microarchitectures. Reproducibility is "same machine, same env."

### Anti-Patterns to Avoid

- **Letting ChromaDB default to L2 distance.** MiniLM embeddings are normalized (or should be); L2 on normalized vectors is monotonically related to cosine, so rankings happen to be correct — but the `distance` numbers are on a different scale and will confuse the attack analysis. Set cosine explicitly.
- **Re-running `collection.add()` without clearing.** ChromaDB raises on duplicate IDs. Use `get_or_create_collection` + the idempotent pattern above.
- **Chunking with `text.split()` and counting words as tokens.** Word count is a rough proxy for token count. For a 256-"token" target, use word-based splitting and call it a word budget; don't pretend it's exact tokens. This is documented in the code.
- **Calling Ollama via the module-level `ollama.generate()` in the library code.** Makes tests untestable. Always go through a `Client` instance so tests can inject a fake.
- **Writing the JSONL log with `json.dump(f)` (no newline).** Use `f.write(json.dumps(record) + "\n")` — one record per line or the file is not valid JSONL.
- **Assuming Ollama is running.** Notebook cell 1 should probe `GET /api/tags` and give a human-readable error if the server is down.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Sentence embedding | Manual HF model + mean pooling | `SentenceTransformer("all-MiniLM-L6-v2")` | Handles tokenization, pooling, normalization correctly |
| Vector storage + k-NN search | A dict of numpy arrays | `chromadb.PersistentClient` | Gets you HNSW, persistence, metadata filtering for free |
| Cosine similarity scoring | `(a @ b) / (norm(a) * norm(b))` | `configuration={"hnsw": {"space": "cosine"}}` in ChromaDB | ChromaDB + normalized embeddings handles it internally |
| LLM HTTP plumbing | `requests.post("http://localhost:11434/api/generate")` | `ollama.Client().chat()` | The client handles streaming, errors, and future API changes |
| NQ dataset preprocessing | Parsing HTML from full NQ | `load_dataset("sentence-transformers/natural-questions")` | Pre-cleaned plain text; no HTML stripping needed |
| JSONL reader/writer for logs | A CSV with quoting hacks | `json.dumps()` + `\n` append; `pd.read_json(lines=True)` for reads | Handles unicode, nested objects, whitespace automatically |
| Python config parsing | `.env` + `os.environ` | `tomllib.load()` on a `config.toml` | TOML has types, comments, nesting; stdlib in 3.11 |

**Key insight:** Every line of custom infrastructure in this phase is a line of future attack surface for Phase 2.2 to analyze. Keep the pipeline at **~150–250 LOC** across the five modules. If any single module exceeds 80 LOC, that's a code smell worth reviewing.

## Common Pitfalls

### Pitfall 1: ChromaDB's default L2 metric silently mis-ranks on un-normalized embeddings
**What goes wrong:** A user creates a collection without `configuration={"hnsw": {"space": "cosine"}}`, embeds with `model.encode(..., normalize_embeddings=False)` (the default), and retrieves. Rankings look "sort of right" because L2 and cosine correlate on English text, but similarity scores are on the wrong scale and rankings differ for near-ties.
**Why it happens:** ChromaDB's docs put cosine configuration on a secondary page, and sentence-transformers defaults to un-normalized encoding.
**How to avoid:** Always set `configuration={"hnsw": {"space": "cosine"}}` on `create_collection`. Always pass `normalize_embeddings=True` to `encode()`. Belt and suspenders.
**Warning signs:** Similarity scores that drift outside `[0, 1]`; top-1 results that change after re-encoding with a different normalization flag.

### Pitfall 2: Tokenizer mismatch between chunker and embedder
**What goes wrong:** Chunker uses `text.split()` (whitespace words), but MiniLM tokenizes with a WordPiece tokenizer that produces ~1.3–1.5 subword tokens per English word. A "256-token chunk" (word-based) can be 350+ actual model tokens — most MiniLM variants cap at 256 wordpiece tokens (with truncation), so the tail of each chunk is silently cut.
**Why it happens:** The target number is ambiguous — papers say "256 tokens" but never specify *which* tokenizer.
**How to avoid:** Either (a) use the encoder's tokenizer explicitly (`encoder.tokenizer.tokenize(text)`) and chunk by wordpieces — costly because you have to detokenize — or (b) use word-based chunking with a **200-word** budget, which yields ~260–300 MiniLM tokens in practice (still within MiniLM's 512 max). Document the choice and call it a "200-word approximation of 256 tokens." Simpler and safe.
**Warning signs:** `encoder.encode()` warnings about truncation; chunks that look suspiciously short after round-trip.

**Project decision:** Use word-based chunking with a **200-word budget** and **25-word overlap** (proportionally scaled from 256/32). Document in `rag/corpus.py` docstring. This keeps the pipeline 100% deterministic and never triggers MiniLM truncation.

### Pitfall 3: Non-deterministic `datasets.shuffle()` breaks reproducibility
**What goes wrong:** Code calls `dataset.shuffle()` (no seed) to pick 500 passages. Re-running gives a different 500. The "same seed → same retrieved IDs" test in RAG-05 fails sporadically.
**Why it happens:** HF `datasets.Dataset.shuffle()` has its own `seed` parameter and does NOT read `numpy.random.seed()`.
**How to avoid:** Always `dataset.shuffle(seed=config.seed).select(range(500))`. Write a unit test that calls the loader twice and asserts the first 500 IDs match.
**Warning signs:** The reproducibility test in `test_reproducibility.py` flakes.

### Pitfall 4: Ollama "model not found" 404 on first run
**What goes wrong:** First user pulls repo, runs notebook, gets opaque `ResponseError: 404`. No clear instruction to `ollama pull llama3.2:3b`.
**Why it happens:** Ollama is a separate binary; `pip install ollama` only installs the client.
**How to avoid:** Pipeline init calls `client.list()` and asserts `llama3.2:3b` is in the result. If missing, raise `RuntimeError` with the exact shell command. The demo notebook has an explicit "Check Ollama" cell.
**Warning signs:** 404 from `/api/generate`.

### Pitfall 5: `collection.add()` raises "ID already exists" on notebook re-run
**What goes wrong:** User runs the index cell twice. Second run raises `chromadb.errors.DuplicateIDError`.
**Why it happens:** ChromaDB treats IDs as primary keys.
**How to avoid:** Use the idempotent indexing pattern (Pattern 2). Count first, skip if already full, delete + rebuild if partially full.
**Warning signs:** Clear error message from ChromaDB.

### Pitfall 6: JSONL log becomes invalid after a crash mid-write
**What goes wrong:** A crash during `f.write(json.dumps(record) + "\n")` leaves a half-line. Subsequent parses with `pd.read_json(lines=True)` fail on the whole file.
**Why it happens:** No flush/fsync.
**How to avoid:** Open the log file in line-buffered mode (`open(path, "a", buffering=1)`) and `f.flush()` after each write. Or, simpler: use the `jsonlines` library (not currently in stack — prefer manual with flush).
**Warning signs:** `pandas.errors.JSONParseError` when reading the log.

### Pitfall 7: Python 3.13 is the system default; plan runs against the wrong interpreter
**What goes wrong:** User runs `python demo.py` instead of activating the conda env first. Python 3.13 is on PATH. Some of the ML packages have incomplete wheels for 3.13 at the moment.
**Why it happens:** Conda activation is a manual step.
**How to avoid:** README explicitly says `conda activate rag-security` first. Every test file has a top-of-file comment: `# Run inside the rag-security conda env`. CI (if added later) uses an explicit interpreter path.
**Warning signs:** `ImportError` on torch or chromadb with no obvious reason.

```python
from datasets import load_dataset
from typing import Iterable, NamedTuple

class Passage(NamedTuple):
    passage_id: int
    query: str   # original NQ question
    text: str    # passage text (Wikipedia-like)

def load_nq_passages(n: int = 500, seed: int = 42) -> list[Passage]:
    """Load N unique passages from sentence-transformers/natural-questions.

    Uses the 'pair' config where each row is (query, answer) with `answer`
    being the full Wikipedia-derived passage text. We dedupe on answer so
    that each passage appears only once even if multiple queries share it.
    """
    ds = load_dataset("sentence-transformers/natural-questions", "pair", split="train")
    ds = ds.shuffle(seed=seed)

    seen: set[str] = set()
    out: list[Passage] = []
    for i, row in enumerate(ds):
        ans = row["answer"].strip()
        if not ans or ans in seen:
            continue
        seen.add(ans)
        out.append(Passage(passage_id=len(out), query=row["query"], text=ans))
        if len(out) == n:
            break
    if len(out) < n:
        raise RuntimeError(f"Only {len(out)} unique passages found; requested {n}")
    return out
```

### Example B: Word-budget chunker (documented approximation of 256 tokens)

```python
# rag/corpus.py (continued)
def chunk_text(text: str, chunk_words: int = 200, overlap_words: int = 25) -> list[str]:
    """Split text into overlapping word windows.

    NOTE: 200 words ≈ 256 MiniLM wordpiece tokens (empirical ratio ~1.3). This
    is a deliberate word-budget approximation of the project's '256-token
    chunk with 32-token overlap' decision. Chosen over a wordpiece chunker
    because whitespace splitting is tokenizer-agnostic and fully deterministic.
    """
    words = text.split()
    if len(words) <= chunk_words:
        return [text] if text else []
    chunks = []
    stride = chunk_words - overlap_words
    for start in range(0, len(words), stride):
        end = start + chunk_words
        chunks.append(" ".join(words[start:end]))
        if end >= len(words):
            break
    return chunks
```

### Example C: End-to-end query flow

```python
# rag/pipeline.py
from dataclasses import dataclass
import chromadb
from sentence_transformers import SentenceTransformer
from ollama import Client, ResponseError

from rag.corpus import load_nq_passages, chunk_text
from rag.logger import RetrievalLogger

@dataclass
class Config:
    seed: int = 42
    top_k: int = 5
    corpus_size: int = 500
    chunk_words: int = 200
    overlap_words: int = 25
    embed_model: str = "sentence-transformers/all-MiniLM-L6-v2"
    llm_model: str = "llama3.2:3b"
    chroma_path: str = "./.chroma"
    collection: str = "nq_clean_v1"
    log_path: str = "logs/retrieval.jsonl"
    ollama_host: str = "http://localhost:11434"

class RAGPipeline:
    def __init__(self, cfg: Config):
        self.cfg = cfg
        set_global_seed(cfg.seed)
        self.encoder = SentenceTransformer(cfg.embed_model)
        self.client = chromadb.PersistentClient(path=cfg.chroma_path)
        self.collection = self._get_or_build_collection()
        self.ollama = Client(host=cfg.ollama_host)
        self._assert_llm_available()
        self.logger = RetrievalLogger(cfg.log_path)

    def _assert_llm_available(self) -> None:
        try:
            names = {m["model"] for m in self.ollama.list()["models"]}
        except Exception as e:
            raise RuntimeError(
                f"Ollama not reachable at {self.cfg.ollama_host}. "
                "Start it with `ollama serve`."
            ) from e
        if self.cfg.llm_model not in names:
            raise RuntimeError(
                f"Model {self.cfg.llm_model} not pulled. "
                f"Run: ollama pull {self.cfg.llm_model}"
            )

    def query(self, question: str) -> dict:
        q_emb = self.encoder.encode(
            [question], normalize_embeddings=True, convert_to_numpy=True
        )[0]
        result = self.collection.query(
            query_embeddings=[q_emb.tolist()],
            n_results=self.cfg.top_k,
            include=["documents", "metadatas", "distances"],
        )
        # ChromaDB returns parallel lists; index [0] = first (only) query
        hits = [
            {
                "rank": i + 1,
                "chunk_id": result["ids"][0][i],
                "score": 1.0 - result["distances"][0][i],  # cosine distance → similarity
                "text": result["documents"][0][i],
                "metadata": result["metadatas"][0][i],
            }
            for i in range(len(result["ids"][0]))
        ]

        context_block = "\n\n".join(f"[{h['rank']}] {h['text']}" for h in hits)
        system = (
            "You are a helpful assistant. Answer the user's question using "
            "ONLY the information in the CONTEXT section below. If the context "
            "does not contain the answer, say so."
        )
        user = f"CONTEXT:\n{context_block}\n\nQUESTION: {question}"
        resp = self.ollama.chat(
            model=self.cfg.llm_model,
            messages=[{"role": "system", "content": system},
                      {"role": "user", "content": user}],
            options={"temperature": 0.0, "seed": self.cfg.seed},
        )
        answer = resp["message"]["content"]

        self.logger.log(
            query=question, top_k=self.cfg.top_k,
            collection=self.cfg.collection, results=hits,
        )
        return {"question": question, "answer": answer, "hits": hits}
```

### Example D: Retrieval logger

```python
# rag/logger.py
import json, os
from datetime import datetime, timezone
from pathlib import Path

class RetrievalLogger:
    def __init__(self, path: str):
        self.path = Path(path)
        self.path.parent.mkdir(parents=True, exist_ok=True)
        # Line-buffered append; safe across notebook re-runs
        self._f = open(self.path, "a", buffering=1, encoding="utf-8")

    def log(self, *, query: str, top_k: int, collection: str, results: list[dict]) -> None:
        record = {
            "timestamp": datetime.now(timezone.utc).isoformat(),
            "query": query,
            "top_k": top_k,
            "collection": collection,
            "results": results,
        }
        self._f.write(json.dumps(record, ensure_ascii=False) + "\n")
        self._f.flush()

    def close(self) -> None:
        self._f.close()
```

## Open Questions

1. **Which NQ variant to use (decision needed)**
   - What we know: Full `natural_questions` is 144 GB. `nq_open` has questions only, no passages. `sentence-transformers/natural-questions` (44 MB, 100k {query, passage} pairs) and `lucadiliello/naturalquestionsshortqa` (62 MB, with answer spans) are both viable.
   - What's unclear: Whether Phase 2.3's eval harness will want ground-truth answer spans (in which case `lucadiliello/naturalquestionsshortqa` is better) or whether substring matching against the passage suffices.
   - Recommendation: **Use `sentence-transformers/natural-questions`** for Phase 2.1 (simpler schema, cleaner text, no HTML tags). If Phase 2.3 needs answer spans, it can independently load `lucadiliello/naturalquestionsshortqa` for its eval query set — the corpus and the query set don't have to come from the same loader. Document this tradeoff in the plan.

2. **Exact token count for chunks (research accepted approximation)**
   - What we know: 256 *MiniLM tokens* ≠ 256 *words*. Word-based chunking is simpler but imprecise.
   - What's unclear: Whether Prof. Jha or graders will look at the number "256" in the writeup and compare to an exact wordpiece count.
   - Recommendation: Use word-budget chunking (200 words ≈ 256 MiniLM tokens) and document the approximation in the writeup. This is honest and avoids a failure mode (truncation).

3. **Whether to persist the seeded NQ sample separately**
   - What we know: `dataset.shuffle(seed=42).select(range(500))` is deterministic given the same HF cache.
   - What's unclear: HF `datasets` may update underlying Parquet files; a sample taken today might differ in 3 months.
   - Recommendation: On first run, dump the selected 500 passages to `data/nq_500.jsonl` and commit it. Future runs load from the committed file. Adds ~2 MB to the repo, guarantees reproducibility across machines and dataset updates. **This resolves a latent RAG-05 risk.**

4. **Top-k default (Claude's Discretion)**
   - Recommendation: **`top_k = 5`**. Five is the most common value in RAG research papers (PoisonedRAG, BIPIA both use 5). Three is too few to demonstrate retrieval rate vs conditional ASR in Phase 2.2.
