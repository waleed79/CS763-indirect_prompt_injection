<decisions>
## Implementation Decisions

### Corpus Dataset
- **D-01:** Use **Natural Questions (NQ)** as the retrieval corpus — Wikipedia-grounded, real user questions, widely used in RAG attack literature (matches PoisonedRAG's setup). Load via HuggingFace `datasets` library (`google-research-datasets/natural_questions` or the `nq_open` variant).
- **D-02:** Index **500 passages** for the initial pipeline. This meets RAG-02 required threshold. Scaling to 1000+ is optional per requirements and deferred until after Phase 2.1 is working.

### Project Structure
- **D-03:** Organize as **Python modules** under a `rag/` package: `rag/pipeline.py`, `rag/corpus.py`, `rag/retriever.py`, `rag/generator.py`, `rag/logger.py`. This allows clean imports from Phase 2.2 (attack module) and Phase 3.1 (defense module).
- **D-04:** Primary **demo/replication entry point** is a `demo.ipynb` Jupyter notebook. The notebook walks through the full pipeline: load corpus → index → query → show retrieved chunks → show LLM answer. Graders can run cell-by-cell. CLI entry point is deferred to Phase 2.2 when batch experiments are needed.

### Corpus Chunking
- **D-05:** Use **fixed-size 256-token chunks** with **32-token overlap**. Simple, deterministic, consistent with what attack papers (including PoisonedRAG) use. One chunk = one document slot, making poisoning ratio reasoning straightforward in Phase 2.2.
- **D-06:** Chunking is done at corpus loading time and stored persistently in ChromaDB — not re-chunked on each query.

### Retrieval Logging
- **D-07:** Log entries are **verbose**: each log record contains query text, top-k doc IDs, cosine similarity scores, full retrieved chunk text, and timestamp. This is required for Phase 2.2 attack analysis (verifying which chunks were retrieved).
- **D-08:** Logs are written to **`logs/retrieval.jsonl`** — one JSON object per line, append mode across runs. Parsed with `pandas` or `jsonlines` for analysis.

### Claude's Discretion
- Top-k default value (3 or 5) — Claude picks what's standard for research RAG pipelines
- Config file format (JSON vs YAML) — Claude picks the simpler option
- ChromaDB collection naming convention — Claude decides
- Exact NQ subset/split to use — Claude picks the most accessible/compatible split

</decisions>

<specifics>
## Specific Ideas

- The `demo.ipynb` notebook should show a visually clear before/after: the query, the retrieved chunks (with scores), and the LLM answer. This is what graders and Prof. Jha will see first.
- ChromaDB's Python client handles persistence automatically — no separate server process needed.
- Ollama must be running locally as a server (`ollama serve`) before the pipeline can generate. The notebook should include a cell that checks Ollama is reachable and prints a clear error if not.
- Random seed + pinned versions + config file are required (RAG-05) — this must be part of Phase 2.1, not an afterthought.

</specifics>
