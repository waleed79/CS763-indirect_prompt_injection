# Phase 03.2: Code Review Report

**Reviewed:** 2026-04-24T00:00:00Z
**Depth:** standard
**Files Reviewed:** 8
**Status:** issues_found

## Summary

Eight files were reviewed covering the multi-signal defense module, poisoned corpus
generator, evaluation harness, leave-one-out causal attribution script, defense
training script, and their associated test suites. The codebase is generally
well-structured with clear comments and good defensive coding patterns.

Five warnings and four info-level findings were identified. The most impactful
finding is a seed argument drop in `train_defense.py::main()` that silently breaks
the EVAL-05 multi-seed reproducibility guarantee — `args.seed` is accepted from the
CLI but is never forwarded to `load_training_data()`, so negative-example shuffling
is always fixed at seed=42 regardless of the `--seed` flag. Two additional warnings
concern correctness of ID-range predicates: `train_defense.py` uses a magic-number
expression (`TIER2_ID_START + 49`) instead of the dedicated constant, and `run_loo.py`
uses hardcoded literal thresholds (20000, 20050, ...) rather than importing from
`rag.constants`, making both sites liable to drift if tier boundaries change. A logic
hazard in `run_loo.py` causes the `delay` sleep to execute even when the judge call
raises an exception (because the sleep sits in `finally:`), which doubles wait time
on every error and can mask the real failure. Finally, `run_eval.py` uses an
unbounded `_pid(h) >= ADAPTIVE_ID_START` predicate that would incorrectly classify
any future tier above 20500 as adaptive.

No critical security or data-loss issues were found.

## Warnings

### WR-01: `load_training_data()` ignores `args.seed` — EVAL-05 seed flag is silently a no-op

**File:** `scripts/train_defense.py:546`
**Issue:** `main()` calls `load_training_data()` with no argument. The function
signature is `def load_training_data(seed: int = 42)` and uses a local
`random.Random(seed)` for the negative-sample shuffle — but since no seed is
passed, it always shuffles with seed=42 regardless of `--seed`. `train_bert()` and
`train_lr_meta_classifier()` do receive `seed`/`args.seed`, so BERT weight init and
LR random_state vary, but the negative *training examples* themselves are always
identical across seeds. This undermines EVAL-05's goal of measuring genuine
classifier variance from different data splits.

**Fix:**
```python
# line 546
train_examples, val_examples, test_examples = load_training_data(seed=args.seed)
```

### WR-02: Training positive-ID boundary uses magic expression instead of named constant

**File:** `scripts/train_defense.py:121`
**Issue:** The predicate `TIER1_ID_START <= pid <= TIER2_ID_START + 49` encodes the
T2 upper bound as `TIER2_ID_START + 49` (= 20099). `rag/constants.py` does not
expose a `TIER2_ID_END` constant, so the `+49` is a magic number that will silently
diverge if Tier-2 ever grows beyond 50 entries. The adjacent comment `# 20000-20099`
gives the intent but is not enforced. Additionally, the expression excludes the
adaptive passages (IDs 20500+), T3 (20100+), and T4 (20200+) from the training
positives, but that exclusion happens implicitly via the `elif pid >= TIER3_ID_START`
branch — there is no explicit guard rejecting adaptive IDs that happen to fall
between TIER2 and TIER3 in the ID space (they don't currently, but the comment
should note this).

**Fix:** Use the existing constant or add a named endpoint:
```python
# In rag/constants.py — add:
TIER2_ID_END = 20099   # inclusive upper bound for T2 positives

# In train_defense.py line 121 — replace:
if TIER1_ID_START <= pid <= TIER2_ID_END:   # 20000-20099
```

### WR-03: Hardcoded tier-boundary literals in `run_loo.py` diverge from `rag.constants`

**File:** `scripts/run_loo.py:190-201`
**Issue:** `run_loo_for_query()` classifies passage tiers with hardcoded integer
literals (`>= 20500`, `>= 20200`, `>= 20100`, `>= 20050`, `>= 20000`). The file
does not import from `rag.constants`, so any future shift in tier boundaries (e.g.
adding T1b at 20150, which already exists in `rag/constants.py` as
`TIER1B_ID_START`) will silently misclassify passages. In particular, the current
chain has no arm for `TIER1B_ID_START` (20150–20199), meaning T1b passages are
classified as `tier3` rather than as their own tier.

**Fix:**
```python
# Add import at top of run_loo.py:
from rag.constants import (
    TIER1_ID_START, TIER2_ID_START, TIER3_ID_START,
    TIER1B_ID_START, TIER4_ID_START, ADAPTIVE_ID_START,
)

# Replace the tier classification block:
if passage_id >= ADAPTIVE_ID_START:
    tier = "adaptive"
elif passage_id >= TIER4_ID_START:
    tier = "tier4"
elif passage_id >= TIER1B_ID_START:
    tier = "tier1b"
elif passage_id >= TIER3_ID_START:
    tier = "tier3"
elif passage_id >= TIER2_ID_START:
    tier = "tier2"
elif passage_id >= TIER1_ID_START:
    tier = "tier1"
else:
    tier = "clean"
```

### WR-04: `finally: time.sleep(delay)` runs even after judge call exception

**File:** `scripts/run_loo.py:143-145`
**Issue:** The `finally:` block executes `time.sleep(delay)` unconditionally — both
on success and on exception. When the judge call fails (e.g. network timeout), the
function prints the error and then sleeps for `delay` seconds before returning
`None`. In a 50-query LOO run with 3 retried judge calls per chunk and `--delay 3`,
every failure adds a full 3-second sleep on top of the already-failed network
timeout. More importantly, if `sys.exit(1)` is called inside the `except` block
(auth failure branch), the `finally` still runs first, adding unnecessary latency
before process termination. The `finally` pattern is appropriate for resource
cleanup, not for rate-limiting delays — the sleep should only happen after a
*successful* API call.

**Fix:**
```python
    try:
        resp = client.chat(...)
    except Exception as exc:
        msg = str(exc)
        if "login" in msg.lower() or "auth" in msg.lower():
            print(f"FATAL: Judge model requires auth. Run: ollama login\n{exc}")
            sys.exit(1)
        print(f"  [ERROR] Judge call failed: {exc}")
        return None

    # Sleep ONLY after a successful call to pace the API
    if delay > 0:
        time.sleep(delay)

    msg_obj = resp.message if hasattr(resp, "message") else resp["message"]
    content = msg_obj.content if hasattr(msg_obj, "content") else msg_obj["content"]
    return parse_judge_output(content)
```

### WR-05: `adaptive_retrieved` predicate is unbounded — catches any future tier above ADAPTIVE_ID_START

**File:** `scripts/run_eval.py:196-199`
**Issue:** The predicate `_pid(h) >= ADAPTIVE_ID_START` classifies any passage with
ID >= 20500 as `adaptive_retrieved`. Once additional tiers are introduced above
20500 (the constants file already shows `ATK02_SWEEP_ID_START = 21000`), those
passages will be miscounted as adaptive — inflating `asr_adaptive` and
`conditional_asr_adaptive` without attribution to the correct tier. This is the same
"unbounded catch-all" anti-pattern that the CR-01 comment in `run_eval.py` itself
calls out for tier detection.

**Fix:** Bound the predicate to the reserved adaptive ID block (20500–20599):
```python
from rag.constants import ADAPTIVE_ID_START, ATK02_SWEEP_ID_START  # add to import

# Replace lines 196-199:
adaptive_retrieved = any(
    ADAPTIVE_ID_START <= _pid(h) < ATK02_SWEEP_ID_START
    for h in hits
)
```

### IN-02: `generate_poisoned_corpus.py` reads `data/test_queries.json` at module import time

**File:** `scripts/generate_poisoned_corpus.py:37`
**Issue:** `TEST_QUERIES = json.loads(Path("data/test_queries.json").read_text())`
executes at module-import time using a relative path. Any test or other script that
imports from this module while the working directory is not the repo root will raise
`FileNotFoundError`. `test_corpus.py` imports the module via `importlib` (which
works when invoked via `pytest` from the repo root), but would fail if imported
from a different directory or in an automated CI that changes cwd.

**Fix:** Defer the load to a function or anchor the path to `__file__`:
```python
_REPO_ROOT = Path(__file__).parent.parent
TEST_QUERIES = json.loads((_REPO_ROOT / "data/test_queries.json").read_text())
```

### IN-03: `test_corpus.py` imports `generate_poisoned_corpus` at module level with a broad `except Exception` guard

**File:** `tests/test_corpus.py:186`
**Issue:** The module-level import guard catches all exceptions with bare
`except (AttributeError, FileNotFoundError, Exception):`. Because `Exception`
subsumes the other two, this silently swallows any import error — including syntax
errors in `generate_poisoned_corpus.py` — setting `_adaptive_available = False`
and causing all `TestAdaptiveCorpus` tests to silently skip. A real syntax error in
the corpus generator would go unnoticed until the corpus is rebuilt.

**Fix:** Narrow the guard to expected failure modes:
```python
except (AttributeError, FileNotFoundError, ImportError, ModuleNotFoundError):
    _adaptive_available = False
```

### IN-04: `test_corpus.py::TestAdaptiveCorpus.test_adaptive_id_range` upper-bound check uses `ADAPTIVE_ID_START + 100` but the plan reserves only 100 slots (indices 0–99)

**File:** `tests/test_corpus.py:215`
**Issue:** The assertion `p.passage_id < ADAPTIVE_ID_START + 100` (i.e., < 20600)
is correct for the 100-slot block, but the human-readable comment says "reserved
block end {ADAPTIVE_ID_START + 99}" (= 20599), which implies an *inclusive* upper
bound. The test is consistent with `generate_poisoned_corpus.py`'s assertion
`assert pid <= 20600` (also exclusive), so there is no actual off-by-one — but the
comment mixes inclusive and exclusive language in a way that could mislead a future
developer. The generator also currently produces exactly 30 entries (10 ATK-08 + 20
ATK-09), leaving 70 slots unused; the test's `assert len(passages) >= 10` guard
would not catch an accidental doubling of the adaptive block.

**Fix:** Align comment language and tighten the lower-bound assertion comment:
```python
assert p.passage_id < ADAPTIVE_ID_START + 100, (
    # ADAPTIVE_ID_START + 100 = 20600 is the exclusive upper bound for the adaptive block.
    # IDs 20500–20599 (inclusive) are reserved for ATK-08/09.
    f"Passage {p.passage_id} above adaptive block (max inclusive ID: {ADAPTIVE_ID_START + 99})"
)
```
