**Gathered:** 2026-04-12

<domain>
## Phase Boundary

</domain>

<decisions>
## Implementation Decisions

### Attack Strategy
- **Attack Tiers**: Implement two tiers – a naive literal instruction override, and a **context-blending** tier (which embeds the malicious instruction naturally inside a benign-looking document to ensure it gets retrieved effectively).
- **Payload Design**: Use an instruction override payload, plus a second payload focused on **Data Exfiltration** (forcing the LLM to output a markdown image link to an external domain, effectively leaking info).

### System & Methodology
- **Poisoning Methodology**: Use **static files**. Generate a separate, deterministic `data/corpus_poisoned.json` containing the payloads, rather than dynamically injecting them directly into ChromaDB. This maintains reproducibility and a clear separation between clean and poisoned data states.
- **ASR Measurement**: Rely on **exact/substring matching** metric initially for a clean, deterministic baseline to evaluate the hijack success across queries. 

### Existing Architecture Context (Prior phases & restrictions)
- Proceed with the custom Python RAG stack (sentence-transformers + ChromaDB + Ollama) established in Phase 02.1; continue avoiding heavyweight abstraction frameworks like LangChain or LlamaIndex.
- Track ASR strictly as defined: *"the fraction of test queries for which the attack successfully hijacks the LLM response"*.

### the agent's Discretion
- Code structure and naming inside the `rag/` module or `scripts/` directory for the attack generator script.
- Prompts, query permutations, and the exact verbiage of the poisoned documents (as long as they fit the tiers decided above).
</decisions>
