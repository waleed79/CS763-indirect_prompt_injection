### Pattern 1: Distinguishing Indirect from Direct Injection (PH1-01)

**What:** The document must explicitly define both types and draw a sharp distinction. This is the single most important conceptual contribution of the problem statement.

**Content specification:**

- **Direct prompt injection:** The user (or someone acting as the user) injects malicious instructions directly into the query input. The attacker IS the user. Defenses: input validation, rate limiting, system prompt hardening.
- **Indirect prompt injection:** The adversary has no access to the user interface. Instead, they poison the knowledge base (retrieval corpus) that the RAG system queries. When a legitimate user asks a benign question, the RAG retrieves attacker-controlled content, which contains embedded instructions that hijack the LLM's response. The attacker is neither the user nor the system operator — they are a third party who can write to the corpus.

**Why this matters for the document:** The course context may conflate the two. Stating the distinction clearly positions the project within the correct threat model and demonstrates conceptual precision that instructors reward.

**Key sentence pattern to use:**
"Unlike direct prompt injection — where the attacker controls the user's input — indirect prompt injection exploits the retrieval boundary: an adversary who can write a single document to the knowledge base can hijack the LLM's response for any user who triggers retrieval of that document."

### Pattern 2: Threat Model Structure (PH1-05, locked by D-01 through D-03)

**Opening scenario (1-2 sentences):**
"Consider an adversary with write access to a document store used by an enterprise RAG-based assistant. Without ever interacting with the LLM, querying the system, or modifying the model weights, the adversary uploads a small number of documents containing embedded instructions."

**Attacker capabilities (bullet list):**
- Corpus write access: can insert, modify, or delete documents in the retrieval database
- Black-box system access: can query the RAG system as a normal user to observe outputs (but cannot access internals)
- Knowledge of the general topic domain: understands what queries users are likely to issue

**Attacker goals (bullet list):**
- Instruction override: force the LLM to follow the attacker's instructions rather than the system prompt
- Payload delivery: exfiltrate user data, produce misinformation, or redirect users to attacker-controlled resources

**Attacker limitations — what they CANNOT control (bullet list):**
- LLM weights or system prompt: the model and its instructions are not accessible
- User queries: the attacker does not know the specific queries users will issue, only the topic domain
- Retrieval mechanism: the attacker cannot deterministically force their document to be retrieved; they rely on semantic similarity between their document and user queries
- LLM inference: the attacker cannot observe intermediate attention weights or generation logits

**Why explicit limitations matter:** Stating what the attacker cannot do is as important as stating what they can. It prevents scope creep and demonstrates rigorous threat modeling. Many student writeups omit attacker limitations entirely.

**Theme 1: RAG Background and Motivation**
- Introduce RAG as the dominant paradigm for grounding LLM outputs in external knowledge

- Cite: Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" (foundational RAG paper — does not need verification, well-known)
- Integration point: "This project's attack surface is precisely the retrieval step that RAG introduces"

**Theme 2: Attack Papers**
- Greshake et al. (2023, arXiv:2302.12173): Established the taxonomy of indirect prompt injection. Their work focused on LLM-integrated applications broadly (browser plugins, email summarizers). This project narrows the focus to RAG-specific corpus poisoning, where the injection vector is the retrieval corpus rather than live web content.
- Zou et al. / PoisonedRAG (2024, arXiv:2402.07867, USENIX Security): Demonstrated that injecting five documents into a million-document corpus achieves 90% ASR. This project builds on this result but adds a defense evaluation that PoisonedRAG does not include.
- Shafran et al. (2025, arXiv:2406.05870, USENIX Security): Showed a different attack variant (denial-of-service via "blocker" documents). Demonstrates breadth of RAG attack surface. Integration point: the injection attacks this project studies are more targeted than jamming.

**Theme 3: Defense and Detection**
- BIPIA (Yi et al., 2023, arXiv:2312.14197, KDD): Proposed four black-box prompt-learning defenses and one white-box fine-tuning defense. Their benchmark covers email QA, web QA, and summarization. Integration point: this project proposes a BERT-based context sanitization classifier operating at the retrieved chunk level — a different architectural locus than BIPIA's prompt-learning defenses.

**Theme 4: Evaluation Frameworks and Industry Context**
- OWASP LLM Top 10 2025 (LLM01:2025): Prompt injection remains the #1 LLM vulnerability for the second consecutive edition. The OWASP definition explicitly covers indirect injection via RAG. Integration point: this project's threat model aligns with LLM01:2025, grounding academic work in industry-recognized risk classification.

## Common Pitfalls

### Pitfall 1: Insufficient Direct/Indirect Distinction
**What goes wrong:** The problem statement mentions "prompt injection" without clearly separating the indirect (corpus-side) variant from direct (user-side). Reviewers who know the field notice immediately.
**Why it happens:** Writers assume the distinction is obvious.
**How to avoid:** Dedicate at least two sentences in Section 3 explicitly to the contrast. Use the phrase "indirect prompt injection" consistently, not just "prompt injection."
**Warning signs:** If the phrase "direct prompt injection" does not appear at all in the document.

### Pitfall 2: Literature Survey Reads as Annotated Bibliography
**What goes wrong:** Each paper gets a short summary paragraph with no connection to this project. Lacks the "thematic synthesis" required by D-04.
**Why it happens:** Easier to write standalone summaries than to synthesize across papers.
**How to avoid:** Organize by theme (D-04), and end each theme with one sentence about what gap this project fills relative to that body of work.

### Pitfall 3: Threat Model Lists Only Capabilities (Omits Limitations)
**What goes wrong:** The threat model bullet list covers what the attacker can do but says nothing about what they cannot access. Graders penalize incomplete threat models.
**Why it happens:** Attackers' limitations are invisible — you have to deliberately think about what is out of scope.
**How to avoid:** Include an explicit "What the attacker cannot control" bullet list (covered in Pattern 2 above).

### Pitfall 4: Execution Plan Is Vague
**What goes wrong:** "Phase 2 will implement the attack module and begin defense work" — no specific deliverables.
**Why it happens:** Writers avoid committing to specifics.
**How to avoid:** List concrete deliverables for each deadline. Use the content from ROADMAP.md (documented in Pattern 4 above).

### Pitfall 5: ArXiv Citations Missing Venue
**What goes wrong:** Citations list only the arXiv preprint number, missing the peer-reviewed publication venue.
**Why it happens:** Papers are found via arXiv search and the venue is not checked.
**How to avoid:** Use the full citation with venue: PoisonedRAG was published at USENIX Security 2024; BIPIA was published at KDD 2025; Greshake et al. appeared at the IEEE S&P Workshop on Security and Privacy in Machine Learning (2023). Always include the venue.
