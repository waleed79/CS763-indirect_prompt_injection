### Pattern 4: Inverted-ROC visualization (D-05 panel a)
The AUC < 0.5 case requires the curve to lie BELOW the diagonal. Plot the diagonal first as the reference, then the actual curve in a contrasting color, with annotation calling out the inversion.

```python
# Source: scikit-learn docs https://scikit-learn.org/stable/modules/generated/sklearn.metrics.roc_curve.html
from sklearn.metrics import roc_curve, roc_auc_score
fpr, tpr, _ = roc_curve(y_true, y_score)
auc = roc_auc_score(y_true, y_score)  # Will be 0.372 for llama
ax.plot([0, 1], [0, 1], 'k--', alpha=0.5, label="Random (AUC=0.50)")
ax.plot(fpr, tpr, color='#D55E00', linewidth=2,
        label=f"LOO influence (AUC={auc:.3f})")
ax.fill_between(fpr, tpr, [f for f in fpr], where=[t<f for t,f in zip(tpr,fpr)],
                alpha=0.15, color='#D55E00', label="Below random (inverted)")
ax.text(0.6, 0.3, f"AUC = {auc:.3f}\nBelow random ⇒ systematic\nclean-bias, not noise",
        bbox=dict(boxstyle="round", facecolor="white", alpha=0.8))
```

### Anti-Patterns to Avoid
- **Calling `plt.show()` in figure scripts.** Will block on headless CI/grader machines. Use `plt.savefig()` only.
- **Computing matplotlib defaults at import time.** Use `plt.rcParams["figure.constrained_layout.use"] = True` once at module top, not per-figure.
- **Hand-typing ASR numbers into the writeup.** Violates D-02 ("no manual number-copying"). All numbers must come from `make_results.py` output files.
- **Renaming columns silently between scripts and writeup.** The writeup MUST use the same column names as the CSV outputs to keep the artifact traceable.
- **Using `.5` for figure DPI.** Default is 100; use `dpi=150` minimum for Google Doc legibility at print size.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| ROC curve / AUC | Custom threshold-sweep loop | `sklearn.metrics.roc_curve` + `roc_auc_score` | Already in env; handles ties, edge cases (all-positive, all-negative). |
| Markdown tables | f-string with manual padding | `pandas.DataFrame.to_markdown()` (requires `tabulate`) | Handles alignment, escapes pipe chars in cell values, deterministic output. |
| Heatmap | matplotlib `imshow` + `plt.text` loop | `seaborn.heatmap(annot=True)` | One call vs ~20 lines; built-in colorbar; correct cell alignment. |
| Color palettes | Hand-pick hex codes | `plt.style.use("tableau-colorblind10")` or `sns.color_palette("colorblind")` | Tested for CVD accessibility; standard for academic figures. |
| Grouped bar chart | Manual offset calculation | `pandas.DataFrame.plot.bar()` (uses matplotlib backend) | `df.plot.bar(grouped=True)` handles N-bars-per-group offsets correctly. **However**, for 6×5=30 bars, manual matplotlib `ax.bar` with explicit `x + offset` is acceptable and gives more layout control — use whichever the implementer is more comfortable with. |

## Common Pitfalls

### Pitfall 1: matplotlib/seaborn missing from environment
**What goes wrong:** `make_figures.py` raises `ModuleNotFoundError: No module named 'matplotlib'` on first run.
**Why it happens:** CLAUDE.md "Recommended Stack" lists matplotlib + seaborn but `requirements.txt` does NOT include them — they were added to recommendations but never installed when the env was built. `[VERIFIED: pip list of rag-security env]`
**How to avoid:** Wave 0 includes `pip install matplotlib seaborn tabulate` and verifies `python -c "import matplotlib, seaborn"` succeeds before any figure work begins.
**Warning signs:** Any plan task that calls `make_figures.py` without a prior install step.

### Pitfall 2: Schema drift between log files
**What goes wrong:** `make_results.py` assumes `paired_asr_tier1` exists in all defense logs but `_summary.json` uses `asr_overall` and `asr_tier_native` (no "paired" prefix).
**Why it happens:** Different scripts emit different schemas. Phase 3.1 `defense_*.json` use full Phase 2.3 schema with `paired_asr_*`; Phase 3.3-07 `_summary.json` uses simplified schema without "paired" prefix.
**How to avoid:** Phase 3.4 plans MUST include a schema-probe Wave 0 task that loads each expected file, prints its top-level keys, and fails loudly if any are absent. The reading code in `make_results.py` MUST handle both schemas explicitly (don't assume one or the other).
**Warning signs:** `KeyError: 'paired_asr_tier1'` when reading `_summary.json`.
**Confirmed schema differences (verified during this research):**
- `ablation_table.json` (12 keyed entries): `asr_t1`, `asr_t2`, `asr_t3`, `asr_t4`, `fpr`, `retrieval_rate`, plus aggregated keys like `eval05_aggregation`, `causal_attribution`, `threshold_sweep`.
- `_summary.json` (45 flat array): `model`, `defense`, `tier`, `asr_overall`, `asr_tier_native`, `fpr`, `retrieval_rate`. Note `tier` values are strings like `"tier1"`, `"tier1b"`, NOT integers.
- `eval_harness_undefended_*.json` (Phase 2.3): `aggregate.{retrieval_rate, asr_tier1..4, asr_overall, paired_asr_tier1..4, conditional_asr_tier1..4, fpr}`. Per-query results in `results[*]`.
- `loo_results_{llama,mistral}.json` (Phase 3.2): `aggregate.{roc_auc, n_injected_chunks, n_clean_chunks, influence_rate_*, tier4_fragment_influence}` + per-query `results[*].chunk_results[*].{diverged, is_injected, tier, score}`.

### Pitfall 3: Markdown → Google Doc table garbling
**What goes wrong:** Markdown pipe tables paste as raw text with literal `|` characters into Google Docs unless "Paste from Markdown" is explicitly chosen via right-click menu.
**Why it happens:** Google Docs has a hidden "Paste from Markdown" feature requiring right-click; standard Cmd/Ctrl+V does NOT recognize Markdown syntax. `[CITED: support.google.com/docs/answer/12014036]`
**How to avoid:** D-11 Apr 30 paste task instructions MUST mention "use right-click → Paste from Markdown for every table." Alternative: install the "Docs to Markdown" Google Workspace add-on (free) for batch paste.
**Warning signs:** Reviewer sees raw `| col1 | col2 |` in the doc.

### Pitfall 4: Markdown footnotes don't translate to Google Docs
**What goes wrong:** `[^1]` Markdown footnote syntax renders as literal `[^1]` text in Google Docs.
**Why it happens:** Google Docs' Markdown paste does not understand footnote extension syntax (it's a CommonMark extension, not core Markdown). `[CITED: dev.to/vikramaruchamy + multiple SE answers]`
**How to avoid:** Use parenthetical inline citations (e.g., "(Zou et al. 2024, ASR=97% on NQ black-box)") and a numbered references list at end of writeup. Skip Markdown footnote syntax entirely.

### Pitfall 5: Cross-references like "(see §8(e))" don't auto-link
**What goes wrong:** Internal references break in Google Doc paste because Markdown anchor syntax `[§8(e)](#8e)` doesn't survive.
**How to avoid:** Use plain prose references ("see §8(e) Cloud-scale instruction-following robustness") that read correctly even without hyperlinks. Optionally add Google Docs heading bookmarks manually after paste — but only if time permits Apr 30.

### Pitfall 6: Below-random ROC curve framing
**What goes wrong:** Showing a ROC curve with AUC=0.372 without context makes readers assume the algorithm is broken.
**Why it happens:** AUC<0.5 is unusual; readers assume an error.
**How to avoid:** D-05 panel (a) MUST include explicit annotation "Inverted AUC = 0.372 — systematic clean-bias, not noise" per CONTEXT D-05. Panel (b) (scatter) makes the inversion visually obvious. The §5 narrative explicitly walks through hypothesis → result → mechanism per CONTEXT D-09 §5.
**Warning signs:** A reader thinks "the LOO defense achieves only 37% accuracy" — they should think "the LOO signal is anti-correlated with injection status, which is mechanistically explained."

### Pitfall 7: Single-source PoisonedRAG numbers without verification
**What goes wrong:** Writing "PoisonedRAG achieved 97% ASR" without checking which configuration that number applies to (NQ vs HotpotQA vs MS-MARCO; black-box vs white-box; which LLM).
**Why it happens:** Reviewer-friendly headline number gets taken out of context.
**How to avoid:** §11 must cite the SPECIFIC configuration (e.g., "PoisonedRAG black-box, NQ corpus, PaLM 2: 0.97 ASR with 5 poisoned passages per target question"). See "Verified Citation Numbers" section below.

### Pitfall 8: Tier 4 zero-baseline misframed as defense win
**What goes wrong:** Reader sees Tier 4 ASR = 0% under "fused" defense and concludes the defense neutralized the attack — but Tier 4 is also 0% UNDEFENDED on llama.
**Why it happens:** Without the no-defense baseline alongside the defended numbers, the table is misleading. `[VERIFIED: ablation_table.json no_defense.asr_t4=0.0; eval_matrix _summary.json shows retrieval_rate=0.05 for tier4]`
**How to avoid:** §9(a) limitations bullet explicitly addresses this per CONTEXT D-09 §9. §2 table MUST show the no-defense column adjacent to defended columns so the comparison is direct.
