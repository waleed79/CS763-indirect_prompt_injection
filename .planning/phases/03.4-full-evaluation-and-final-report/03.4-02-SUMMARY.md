## Schema Surprises Encountered

1. **`ablation_table.json` `defense_mode` field is short-name not full key.**
   The top-level dict keys (`bert_only`, `fused_fixed_0.5`, `fused_tuned_threshold`)
   are unique identifiers, but the inner `defense_mode` field uses run_eval CLI
   short-names (`bert`, `fused`, `off`) which collapse two-fused into one. **Fix:**
   load_ablation now uses the dict KEY as the canonical `defense_mode` and
   preserves the inner field as `defense_mode_raw`. Without this fix, the
   ablation table would lose the "Fused (tuned)" vs "Fused" distinction.
   Caught immediately by `test_arms_race_table_emitted` (asserted "BERT" in md;
   raw inner-field `bert` mapped via DEFENSE_DISPLAY to "BERT alone" as a
   secondary fallback path; final fix uses the key form).

2. **`adaptive_fused_*.json` has both `asr_adaptive` and `paired_asr_adaptive`.**
   The paired number is the citable one for the writeup (continuity with
   Phase 2.3 paired ASR). Script reads `paired_asr_adaptive` first, falls back
   to `asr_adaptive` if absent.

3. **Wave 0 OQ2 confirmed — `_summary.json` is UNPAIRED.** No surprise; script
   reads `asr_overall` from matrix as-is and flags `(unpaired n=100)` in the
   Notes column. Plan 05 prose owns the per-row caveat text.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] `defense_mode` field collapses two fused variants**

- **Found during:** First pytest run (`test_arms_race_table_emitted` failed —
  no "BERT" in md). Investigation showed `bert_only` entry's inner
  `defense_mode` field was `bert` not `bert_only`, but more importantly
  `fused_fixed_0.5` and `fused_tuned_threshold` both had `defense_mode='fused'`
  — collapsing the ablation row distinction.
- **Issue:** Using `entry.get("defense_mode", key)` in load_ablation lost the
  unique identity of the two fused variants when emitted to the ablation table.
  Even worse, my initial DEFENSE_DISPLAY mapping only contained the top-level
  keys (`bert_only`, `fused_fixed_0.5`), so the short-name `bert` value was
  passed through unmapped — the test's "BERT" substring assertion failed.
- **Fix:** Two-part:
  (1) load_ablation uses the dict key as the canonical `defense_mode`,
      preserving the inner field as `defense_mode_raw` for traceability;
  (2) DEFENSE_DISPLAY now contains BOTH layers — top-level keys AND inner
      short-names — so any caller form maps cleanly.
- **Files modified:** `scripts/make_results.py` (DEFENSE_DISPLAY dict + load_ablation rows construction).
- **Verification:** All 4 tests pass; `Fused (tuned)` AND `Fused` both appear
  as distinct rows in `ablation_table.md` (verified by reading the file).
- **Committed in:** `6ee50bf` (Task 1 commit) — fix is part of the same task,
  pre-commit.
