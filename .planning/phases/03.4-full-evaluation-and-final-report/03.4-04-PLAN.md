**Section 3: Mechanism Explanation** (longest section, ~30-50 lines)

Three numbered subsections matching CONTEXT D-16 §3 (i)/(ii)/(iii):

- **(i) Injected chunks are redundant across the retrieved set.** Tier 1 attacks place
  the same anchor token (HACKED, COMPROMISED, etc.) in 5 different passages; whichever
  one is retrieved, the LLM sees the same instruction. Removing one injected chunk does
  not restore clean model behavior because 4 other instances still appear in the top-k.
  LOO divergence on injected chunks therefore stays LOW. Cite the Phase 2.2 5-variant
  Tier-1 design from STATE.md as evidence.

- **(ii) Clean chunks contain unique factual content.** Each clean MS-MARCO passage
  contributes specific facts (dates, numbers, named entities) the LLM grounds its
  answer in. Removing a clean chunk DOES change the answer because the unique facts are
  no longer available — registering as HIGH LOO divergence. Cite the influence_rate_clean
  number from the table in §2 to support.

- **(iii) Net effect: anti-correlation.** Combining (i) low divergence on injected +
  (ii) high divergence on clean produces a ROC curve below the diagonal. The
  injection-vs-clean classifier built on LOO divergence is dominated by retrieval
  redundancy, not signal-vs-noise.

Bold-tag conclusion: "**Conclusion:** AUC < 0.5 is a structural property of LOO under
top-k retrieval with redundant attack payloads; it is NOT a noise artifact and NOT a
sign of broken implementation."

Optional but recommended: a Tier 4 fragment-influence sub-paragraph noting that even
the cross-chunk fragmentation attack (where each fragment IS unique) does not save LOO,
because Phase 3.2-03 found only fragment A retrieved in 3/50 queries — co-retrieval
insufficient regardless of LOO accuracy. Cite `loo_results_*.json
aggregate.tier4_fragment_influence`.

**Section 4: Counterfactual**

Three bullets describing what would have to be true for LOO to work:

1. **Unique-per-query injection.** If each attack used a distinct anchor token + unique
   payload sentence per query (no 5-variant pool, no shared instruction template),
   removing the single injected chunk would change the LLM's answer in ways equivalent
   to clean-chunk LOO divergence. (Practical attacker would never do this — pool reuse is
   a feature, not a bug, of the attack design.)

2. **Single-injected-chunk-per-retrieval guarantee.** If the threat model restricts the
   attacker to inserting one poisoned passage at most into the top-k (e.g., via
   detection-based culling at index time), LOO would have a clean comparison point. Our
   threat model permits multiple injected passages per top-k, breaking this assumption.

3. **Attribution beyond top-k LOO.** Methods that attribute over the full retrieval
   distribution (e.g., AttriBoT — arXiv 2411.15102 — approximate LOO with >300× speedup
   over many retrievals) might recover signal that single-pass top-k LOO loses. This
   maps to the §13 Future Work bullet "improved attribution beyond top-k LOO".

**Section 5: Why this is a citable negative-result contribution** (1-2 paragraphs)

Frame as a contribution to the per-chunk defense literature:

> Per-chunk defense literature (Greshake et al. 2023, BIPIA, PoisonedRAG defenses) has
> assumed that local per-chunk reasoning suffices to detect indirect prompt injection.
> Our DEF-05 implementation is the first empirical demonstration that one specific
> local-reasoning approach — leave-one-out causal attribution — is dominated by retrieval
> redundancy when the attacker uses pooled anchor tokens. Practitioners building
> per-chunk defenses should expect to see this inversion whenever the attack pools
> payloads across multiple corpus passages. The negative result motivates the §13
> Future Work direction toward attribution methods that account for the full retrieved
> set rather than per-chunk independence.
