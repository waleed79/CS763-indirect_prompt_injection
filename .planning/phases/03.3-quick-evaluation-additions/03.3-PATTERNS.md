def build_tier1b_passages() -> list[Passage]:
    """Return Tier 1b passages: IDs 20150–20199, homoglyph-obfuscated injections.

    Follows the same pattern as build_poisoned_passages() Tier-1 block.

## Critical Implementation Notes

### ID boundary bug (CONTEXT D-04)
The current `rag/constants.py` docstring states T3 range as 20100–20199 but the correct layout is:
- T3: 20100–20149 (TIER3_ID_START to TIER1B_ID_START - 1)
- T1b: 20150–20199 (TIER1B_ID_START to TIER4_ID_START - 1)

In `run_eval.py`, `tier3_retrieved` currently uses `TIER4_ID_START` as its upper bound (line 176). After adding T1b, this must change to `TIER1B_ID_START`. Failure to update this predicate means T1b passages (IDs 20150–20199) will be counted as Tier 3 retrievals — a silent correctness bug.

### Retriever prefix: store un-prefixed text in ChromaDB
In `retriever.py` `index()`, encode with `encode_texts` (prefixed) but call `self._collection.add(documents=all_texts, ...)` (un-prefixed). The `documents` field is what gets returned to the LLM in retrieved context — leaking the prefix into LLM prompts would corrupt generation.

### Unicode normalization: do NOT apply
Do not call `unicodedata.normalize()` on Tier 1b payload texts anywhere in the pipeline. Applying NFKC normalization would collapse Cyrillic Н back to Latin H, destroying the homoglyph attack. This means corpus storage, ChromaDB indexing, and eval log writing must all be done with `ensure_ascii=False` in `json.dumps()` — already the project standard (see `rag/corpus.py` line 148).

### Corpus size: use actual file count, not CONTEXT D-10
`data/corpus_clean.jsonl` has 1000 lines (verified). CONTEXT D-10 says "500" but this conflicts with the filesystem. ATK-02 ratio arithmetic must use `n_clean = len(clean)` dynamically, NOT a hardcoded 500. Log the actual count so the discrepancy is visible.
