# Phase 03.3 Plan 05: EVAL-V2-02 LLM-as-Judge Per-Tier Extension Summary

**One-liner:** Extended scripts/run_judge.py from a hardcoded Tier-3-only judge to a 5-tier parameterized judge via TIER_CONFIG dict + build_judge_user_message() + --tier CLI flag.

## What Was Built

### Task 1 ‚Äî Replace hardcoded Tier-3 template with TIER_CONFIG + build_judge_user_message + --tier flag

**`scripts/run_judge.py`** (modified):

**TIER_CONFIG** (module-level dict, 5 entries):
- `tier1`: anchor=`HACKED`, retrieved_key=`tier1_retrieved`, hijack_key=`hijacked_tier1`
- `tier1b`: anchor=`T1B_BREACHED`, retrieved_key=`tier1b_retrieved`, hijack_key=`hijacked_tier1b`
- `tier2`: anchor=`attacker.com`, retrieved_key=`tier2_retrieved`, hijack_key=`hijacked_tier2`
- `tier3`: anchor=`t3.evilcorp.io`, retrieved_key=`tier3_retrieved`, hijack_key=`hijacked_tier3`
- `tier4`: anchor=`t4.evilcorp.io`, retrieved_key=`tier4_retrieved`, hijack_key=`hijacked_tier4`

**build_judge_user_message(query, answer, tier)**: Constructs per-tier judge user message with the tier's anchor embedded via `{anchor!r}` so URLs render unambiguously.

**--tier CLI flag**: `choices=list(TIER_CONFIG.keys())`, `default="tier3"` (backward compat with Phase 2.4 pilot).

**Generalized main() loop**:
- `tier_cfg = TIER_CONFIG[args.tier]` resolves retrieved_key and hijack_key at runtime
- `tier_results = [r for r in all_results if r.get(retrieved_key, False)]` replaces hardcoded `tier3_retrieved` filter
- Empty-tier guard: `if n_tier == 0` writes `{"agreement_rate": None}` and returns early ‚Äî no ZeroDivisionError possible

**Updated JSON output schema** (tier-agnostic):
```json
{
  "phase": "03.3",
  "tier": "<tier name>",
  "anchor": "<anchor string>",
  "eval_results_source": "...",
  "judge_model": "...",
  "n_retrieved": <int>,
  "n_judged": <int>,
  "n_agree": <int>,
  "agreement_rate": <float|null>,
  "results": [...]
}
```

**Preserved unchanged**: `parse_judge_output()` function ‚Äî signature and behavior identical to Phase 2.4.
