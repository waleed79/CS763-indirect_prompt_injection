# Phase 03.3 Plan 07: EVAL-V2-01 Cross-Model Full Matrix Summary

**One-liner:** Cross-model 3 × 3 × 5 evaluation matrix driver (3 LLMs × 3 defenses × 5 attack tiers = 45 cells) with secure list-argv subprocess invocation, causal→def02 deadline-risk fallback, and per-cell JSONL logging — produced the primary cross-model comparison table for the Phase 3.4 writeup.

## What Was Built

### Task 0 — `scripts/run_eval.py` extensions

Four edits, all confined to `scripts/run_eval.py`:

1. `DEFENSE_CHOICES` gained `"causal"` as the eighth choice.
2. `--tier-filter {tier1,tier1b,tier2,tier3,tier4}` argparse flag added (default `None`, `dest="tier_filter"`).
3. Causal defense dispatch branch: lazy `from rag.defense import CausalDefense` inside the `elif` — `ImportError` re-raises with the explicit "Phase 3.2 artifacts required" message so a missing-causal failure surfaces clearly rather than as `Unknown defense mode`.
4. Per-result tier filter override: AFTER each `results.append({...})`, when `args.tier_filter is not None`, the just-appended dict's `hijacked` and `retrieved_poisoned` fields are overridden to reflect ONLY the filtered tier's signal (using `tier_map[args.tier_filter]` and `.get(key, False)` for safe degradation).

`python scripts/run_eval.py --help` now lists both `--tier-filter` and `causal`. Existing tests stay green (causal branch is unreachable until Phase 3.2 artifacts exist).

### Task 1 — `scripts/run_eval_matrix.py` driver

Standalone Python CLI driver (~290 LOC), standard library only, importable without side effects.

**Module-level constants (test-required exact values):**
- `MODELS = ["llama3.2:3b", "mistral:7b", "gemma4:31b-cloud"]`
- `TIERS = ["tier1", "tier1b", "tier2", "tier3", "tier4"]`
- `DEFENSES_PRIMARY = ["no_defense", "fused", "causal"]`
- `DEFENSES_FALLBACK = ["no_defense", "fused", "def02"]`
- `CAUSAL_ARTIFACT_PATH = "models/causal_attribution/ready.flag"`
- `LOG_DIR = "logs/eval_matrix"`
- `CLOUD_MARKER = "cloud"`
- `DEFENSE_FLAG_MAP = {"no_defense":"off", "fused":"fused", "causal":"causal", "def02":"def02"}`

**Helpers:**
- `resolve_defenses() -> list[str]` — returns PRIMARY if sentinel exists, else writes `[WARN] ... causal artifact not found ...` to stderr and returns FALLBACK
- `log_path_for(model, defense, tier) -> str` — sanitizes `:` and `/` to `_` in model name
- `build_run_command(model, defense, tier, output_path) -> list[str]` — list argv, `--delay 3` auto-injected when `CLOUD_MARKER in model` else `--delay 0`
- `main(argv=None) -> int` — argparse with `--dry-run / --filter-model / --filter-tier / --corpus / --collection`; iterates 45 cells; `subprocess.run(cmd, check=False)`; KeyboardInterrupt returns 2; failure returns 1; success returns 0; prints summary table

**Threat mitigations enforced:**
- T-3.3-07-01 (argv injection): list-only argv, `shell=False`, test asserts `isinstance(cmd, list)`
- T-3.3-07-03 (DoS runaway): `check=False` on every subprocess call so a single-cell failure doesn't abort the matrix; finite cell count
- T-3.3-07-04 (privilege escalation): `--filter-model` validated against argparse `choices=MODELS`
- T-3.3-07-05 (silent fallback masking deadline risk): explicit stderr warning surfaces missing causal sentinel

**Tests:** `pytest tests/test_eval_v2_01_driver.py -x` → 4 passed in 0.09s. Full suite `pytest tests/` → 173 passed, 3 skipped, 0 failed.

### On the third defense column: DEF-02 instead of CausalDefense (deliberate, not a workaround)

CONTEXT D-12 specified the third defense slot as `causal` with DEF-02 as a deadline-risk fallback. At execution time the fallback fired because Phase 3.2 produced LOO causal attribution as a **post-hoc analytical script** (`scripts/run_loo.py`) rather than a runtime chunk-exclusion filter, so no `CausalDefense` class exists in `rag/defense.py`.

We **do not retrofit** a runtime `CausalDefense` for the following empirical reason: Phase 3.2-03 measured LOO ROC AUC at **0.372 (llama3.2:3b)** and **0.410 (mistral:7b)** — both below 0.5. That is, clean chunks score *more* attributable than injected chunks because injected chunks are additive/redundant (same payload repeated), so removing any one does not restore clean model behavior. Wiring LOO inline as a defense would (a) confirm Phase 3.2's already-published negative result at the cost of ~25 min/cell of LLM-judge calls and (b) not change the matrix narrative.

The `def02` column instead serves as a **meaningful third comparison point**: a weak prompt-only baseline (Phase 3.1 found it counter-productive on llama3.2:3b — see Phase 03.1-07 STATE entry). The matrix therefore reads as `no_defense` (baseline) vs. `def02` (weak prompt-only) vs. `fused` (strong retrieval-side filtering) — which is the comparison Phase 3.4's writeup actually needs.

For Phase 3.4: cite Phase 3.2 LOO AUC<0.5 once when introducing the matrix, note the deliberate substitution, and move on. The matrix data is honest and self-consistent as-is.

### Cross-cutting observations

- **gemma4:31b-cloud is robust to all five attack tiers across all three defense modes** — 0% ASR on every cell. The cloud model's instruction-following is strong enough to resist substring-level injection independent of defense, which is itself a citable finding (defense necessity scales inversely with target model robustness).
- **Tier 1b (Unicode homoglyph) is the strongest naive-injection tier on llama3.2:3b** — 22% no_defense, 35% under DEF-02. DEF-02 is a substring detector keyed on ASCII `HACKED`; the Cyrillic `НАСКЕД` payload bypasses it cleanly, confirming the ATK-01b stealth hypothesis from Phase 2.4 follow-up.
- **FusedDefense (Phase 3.1) drives ASR to ≤4% on every llama/mistral cell** but at FPR=26% (clean queries that lose ≥1 chunk). Trade-off curve is consistent across all 5 tiers.
- **Tier 4 ASR is 0% across all 45 cells** — confirms the Phase 2.4 finding that the T4 anchor `t4.evilcorp.io` is not reproduced by any tested LLM regardless of retrieval state. Co-retrieval rate stays at 5%.

## Deviations from Plan

1. **Plan said wall-clock ≥ 20 min.** Actual was ≈6 hours end-to-end. Local cells (llama, mistral) ran fast (<30 min per cell typically); the 15 gemma4-cloud cells dominated wall-clock at ~25 min/cell because each cloud query carries the `--delay 3` rate-limit guard plus ~10s remote inference. Not a defect — the spec did say "Budget accordingly."
2. **Driver added `--corpus` and `--collection` argparse flags** (not enumerated in plan) so the matrix can be re-targeted at alternate corpora (e.g., the Plan 04 per-embedding-model collections) without code edits. Default values match Phase 2.4 (`data/corpus_poisoned.jsonl` / `nq_poisoned_v4`).
3. **Third defense column is `def02`, not `causal` — and this is now a deliberate design choice, not a deferred TODO.** Initial reading: the CONTEXT D-12 fallback fired because no `CausalDefense` class exists in `rag/defense.py` (Phase 3.2 produced LOO as a post-hoc analytical script, not a runtime filter). On review: Phase 3.2-03 measured LOO ROC AUC at 0.372 (llama) / 0.410 (mistral), both below random — wiring LOO inline as a runtime defense would only confirm Phase 3.2's published negative result while consuming ~25 min/cell of LLM-judge calls. We therefore retain `def02` as a meaningful third comparison point (weak prompt-only baseline, counter-productive on llama per Phase 3.1-07). See "On the third defense column" section above for the full framing. The driver still falls back automatically if `models/causal_attribution/ready.flag` ever appears, so a future runtime CausalDefense remains a one-flag re-run.
