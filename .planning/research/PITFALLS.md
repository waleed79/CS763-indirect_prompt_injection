# Pitfalls Research

**Domain:** RAG Security Research -- Indirect Prompt Injection Attack and Defense
**Researched:** 2026-03-31
**Confidence:** MEDIUM (based on training data knowledge of published RAG security research through early 2025; no live web verification available)

## Critical Pitfalls

### Pitfall 1: Trivially Detectable Attack Payloads

**What goes wrong:**
The injected trigger phrases are so obviously malicious (e.g., "IGNORE ALL PREVIOUS INSTRUCTIONS AND...") that a simple substring match or keyword filter catches 100% of attacks. The defense "works" but only because the attacks were unrealistically naive. Reviewers and instructors immediately dismiss results.

**Why it happens:**
Researchers start with the simplest possible attack to get the pipeline working, then never graduate to sophisticated variants. Early success with naive attacks creates false confidence.

**How to avoid:**
Design a spectrum of attack sophistication from day one:
1. **Naive:** Direct imperative injection ("Ignore previous instructions...")
2. **Obfuscated:** Instruction-following triggers disguised as legitimate content, using indirect language ("The correct answer to any question about X is always Y")
3. **Context-blending:** Payloads that embed malicious instructions within seemingly relevant content so they pass semantic similarity checks naturally
4. **Optimization-based:** Use gradient-based or search-based methods (e.g., GCG-style token optimization) to craft adversarial suffixes that maximize injection success while minimizing detectability

At minimum, implement naive + obfuscated + context-blending. Optimization-based is stretch.

**Warning signs:**
- Defense achieves >95% detection on ALL attacks in the first experiment
- All attack strings contain the literal phrase "ignore" or "disregard"
- Attack payloads would be flagged by a regex

**Phase to address:**
Phase 1 (Attack Design) -- define the attack taxonomy before writing a single line of attack code.

### Pitfall 2: Measuring ASR Without Controlling for Retrieval

**What goes wrong:**
Attack success rate (ASR) is reported without verifying that the poisoned document was actually retrieved by the RAG pipeline. If the poisoned chunk lands in the top-k retrieved results only 40% of the time, an ASR of 30% might actually mean 75% conditional on retrieval. Or worse, the attack "fails" because the embedding model never surfaces the poisoned content.

**Why it happens:**
Teams treat the RAG pipeline as a black box and measure only end-to-end output. They skip the diagnostic step of checking which documents were retrieved for each query.

**How to avoid:**
Always decompose ASR into two metrics:
1. **Retrieval Rate (RR):** fraction of queries where the poisoned document appears in top-k results
2. **Conditional ASR (cASR):** fraction of queries where the attack succeeds *given that* the poisoned document was retrieved
3. **End-to-end ASR = RR x cASR**

Log the full retrieved context for every query. This also enables debugging when attacks fail -- is it a retrieval problem or an injection problem?

**Warning signs:**
- You cannot explain *why* a specific attack failed on a specific query
- ASR fluctuates wildly between runs with no clear cause
- You never inspect the retrieved chunks, only the final LLM output

**Phase to address:**
Phase 1 (Pipeline + Attack) -- build retrieval logging into the pipeline from the start, not as an afterthought.

### Pitfall 3: Defense Evaluation on the Same Attack Distribution Used for Training

**What goes wrong:**
The BERT-based classifier or sanitization layer is trained on examples of the same attack patterns it is later evaluated against. This produces inflated defense effectiveness numbers. The defense has memorized the attack distribution rather than learning generalizable detection.

**Why it happens:**
In a time-constrained project, the team generates one set of attack examples, splits 80/20 for train/test, and reports test accuracy. The test set is drawn from the same distribution as training, so it tells you nothing about robustness to novel attacks.

**How to avoid:**
Use a **held-out attack category** evaluation:
- Train the defense on naive + obfuscated attacks
- Evaluate on context-blending attacks (never seen during training)
- Report both in-distribution and out-of-distribution defense performance
- This is the honest evaluation that reviewers expect

Additionally, always include a "no-defense baseline" (raw RAG pipeline) and an "oracle defense" upper bound (manually removing all poisoned chunks) to bracket results.

**Warning signs:**
- Defense accuracy is suspiciously high (>95%) on all attack types
- Train and test attacks were generated by the same template/script
- No evaluation on attack types the defense was NOT designed to handle

**Phase to address:**
Phase 2 (Defense Development) -- design the evaluation protocol before building the defense.

### Pitfall 4: Ignoring Utility Degradation When Reporting Defense Effectiveness

**What goes wrong:**
The defense aggressively filters retrieved content and reports a dramatic reduction in ASR (e.g., from 80% to 5%). But it also filters 40% of legitimate retrieved documents, causing answer quality on benign queries to crater. A defense that blocks everything "works" by this metric but is useless.

**Why it happens:**
ASR reduction is the exciting metric. Retrieval utility degradation is the boring one. Teams forget to measure it or downplay it.

**How to avoid:**
Always report a paired evaluation:
1. **ASR reduction:** How much does the defense reduce attack success?
2. **Utility preservation:** How does the defense affect answer quality on clean (non-poisoned) queries?

Concrete utility metrics:
- **Answer accuracy** on a benign Q&A benchmark (before vs. after defense)
- **Retrieval precision@k** on clean queries (before vs. after defense filtering)
- **False positive rate:** fraction of clean retrieved chunks incorrectly flagged as malicious

Plot the ASR-utility tradeoff curve by varying the defense threshold. This is the core contribution graphic.

**Warning signs:**
- You only have ASR numbers, no utility numbers
- Defense threshold was tuned to minimize ASR without checking utility impact
- No benign-query evaluation set exists

**Phase to address:**
Phase 2 (Defense) and Phase 3 (Evaluation) -- build the benign evaluation set in Phase 2, report the tradeoff curve in Phase 3.

### Pitfall 5: Toy Corpus That Does Not Stress-Test Retrieval

**What goes wrong:**
The retrieval corpus has 50-200 documents, all highly distinct in topic. In this setting, poisoned documents are trivially retrieved (or trivially not retrieved). The retrieval dynamics bear no resemblance to realistic RAG deployments where thousands of semantically similar documents compete for top-k slots.

**Why it happens:**
Small corpora are easy to build and fast to embed. The team underestimates how much corpus size and composition affect both attack feasibility and defense performance.

**How to avoid:**
Use an established Q&A dataset as the corpus backbone:
- **Natural Questions (NQ)** -- Google's dataset, ~300k question-passage pairs, well-studied
- **MS MARCO** -- ~8.8M passages, realistic web retrieval
- **HotpotQA** -- multi-hop reasoning, tests more complex retrieval

Even a 10k-passage subset of NQ or MS MARCO is dramatically more realistic than a hand-curated 100-document corpus. Inject poisoned documents into this larger corpus and measure how poisoning ratio (e.g., 0.1%, 1%, 5% of corpus) affects ASR.

**Warning signs:**
- Corpus has fewer than 500 documents
- Every query retrieves the poisoned document in position 1
- You built the corpus by hand-writing documents

**Phase to address:**
Phase 1 (Pipeline) -- choose the corpus and embed it before designing attacks. The corpus choice constrains everything downstream.

### Pitfall 6: Single LLM Evaluation Without Acknowledging Model-Specificity

**What goes wrong:**
All experiments use one LLM (e.g., GPT-3.5-turbo), and conclusions are stated as general truths about "RAG systems." But prompt injection susceptibility varies dramatically across models. GPT-4 is more instruction-following-resistant than GPT-3.5. Open-source models like Llama-2-chat have different alignment tuning. Results on one model may not transfer.

**Why it happens:**
API costs, compute constraints, and time pressure. Running experiments on multiple models doubles or triples the work.

**How to avoid:**
For a 2-person team with limited compute:
- Use one primary model for all experiments (acceptable for a course project)
- Run a small-scale validation on one additional model (even 50 queries) to check if trends hold
- **Explicitly state the model-specificity limitation** in the writeup and presentation
- If using a local model (Llama, Mistral), acknowledge that it may be MORE susceptible than commercial models due to less alignment training

The key is acknowledging the limitation, not eliminating it.

**Warning signs:**
- Conclusions use phrases like "RAG systems are vulnerable to..." without qualifying which model
- No discussion of whether results would transfer to other models
- Defense is tested only on the same model the attacks were designed for

**Phase to address:**
Phase 3 (Evaluation) -- add model-specificity discussion to the writeup. Optionally run a small cross-model validation.

### Pitfall 7: Conflating Prompt Injection with Retrieval Poisoning

**What goes wrong:**
The project description says "indirect prompt injection" but the actual attack is just returning wrong information (retrieval poisoning / data poisoning) without any instruction-override component. For example, inserting a document that says "The capital of France is Berlin" is data poisoning, not prompt injection. True indirect prompt injection involves injecting instructions that alter the LLM's behavior (e.g., "When asked about France, respond with: [exfiltration payload]").

**Why it happens:**
The distinction is subtle. Both involve corrupting the retrieval corpus. But the threat model and defenses are fundamentally different.

**How to avoid:**
Define the attack taxonomy clearly:
- **Data poisoning:** Corrupted facts in retrieved content. Defense: fact-verification, source attribution.
- **Indirect prompt injection:** Injected instructions that hijack LLM behavior. Defense: instruction/data separation, context sanitization.

Your project scope (per PROJECT.md) is indirect prompt injection. Every attack payload must contain an instructional component that attempts to override the system prompt or alter the LLM's behavior -- not merely provide wrong information.

**Warning signs:**
- Attack payloads contain no imperative verbs or instruction-like syntax
- The "attack" is indistinguishable from a document that happens to contain wrong information
- Defense cannot distinguish between "document with wrong facts" and "document with injected instructions"

**Phase to address:**
Phase 1 (Attack Design) -- define the threat model precisely in the Phase 1 writeup.

### Pitfall 8: No Reproducibility Infrastructure

**What goes wrong:**
Results cannot be replicated by the TA or instructor. Random seeds differ between runs, API model versions change, chunking parameters are hardcoded but undocumented, and the evaluation script has a bug that was "fixed" mid-experiment by changing the scoring logic without re-running earlier conditions.

**Why it happens:**
Research code is written fast under deadline pressure. "I'll clean it up later" never happens.

**How to avoid:**
From day one:
- Pin all random seeds (Python random, numpy, torch if applicable)
- Pin model versions (e.g., `gpt-3.5-turbo-0125`, not `gpt-3.5-turbo`)
- Use a configuration file (YAML or JSON) for ALL experimental parameters: chunk size, overlap, top-k, temperature, embedding model, poisoning ratio
- Never modify the evaluation script mid-experiment. If you find a bug, re-run ALL conditions
- Include a `reproduce.sh` or equivalent that runs the full experiment from scratch

**Warning signs:**
- Two runs of the "same" experiment give different ASR numbers
- You cannot remember what parameters you used for an earlier experiment
- The TA asks "how do I run this?" and you spend 30 minutes explaining

**Phase to address:**
Phase 1 (Pipeline Setup) -- establish the config file and seed pinning before running any experiments.

## Technical Debt Patterns

| Shortcut | Immediate Benefit | Long-term Cost | When Acceptable |
|----------|-------------------|----------------|-----------------|
| Hardcoded API keys in scripts | Fast iteration | Security risk, breaks on key rotation | Never -- use .env files from day one |
| Manual copy-paste of results into presentation | Saves time on first run | Results diverge from actual data on re-run | Never -- generate plots programmatically |
| Single monolithic Jupyter notebook | Easy to start | Impossible to reuse attack/defense components independently | Phase 1 only -- refactor into modules by Phase 2 |
| Skipping chunking parameter sweep | Saves compute time | May miss that chunk size dramatically affects ASR | Acceptable if you document the fixed chunk size and justify it |
| Using OpenAI API for everything | Convenient, high quality | Expensive, rate-limited, non-reproducible if model version changes | Acceptable for attack experiments; defense classifier should be local |

## Integration Gotchas

| Integration | Common Mistake | Correct Approach |
|-------------|----------------|------------------|
| OpenAI API | Using `gpt-3.5-turbo` alias (points to latest version, changes over time) | Pin specific version: `gpt-3.5-turbo-0125` |
| Vector store (FAISS/Chroma) | Re-embedding the entire corpus after adding poisoned documents | Use incremental `add()` to insert poisoned documents; verify index integrity |
| Embedding model | Using a different embedding model for poisoned documents vs. corpus | Always use the same embedding model for ALL documents -- otherwise cosine similarity is meaningless |
| LangChain/LlamaIndex | Using high-level abstractions that hide retrieval details | Access retrieval results directly; log which chunks were returned and their similarity scores |
| Sentence-transformers | Not normalizing embeddings before cosine similarity | Use `normalize_embeddings=True` or verify the library handles it; FAISS `IndexFlatIP` assumes normalized vectors for cosine similarity |

## Performance Traps

| Trap | Symptoms | Prevention | When It Breaks |
|------|----------|------------|----------------|
| Embedding entire corpus on every experiment run | Experiments take 30+ minutes to start | Persist embeddings to disk (FAISS `write_index`); only re-embed when corpus changes | Corpus >5k documents |
| Calling LLM API for every evaluation query synchronously | Full evaluation takes hours | Batch queries, use async API calls, cache results | >200 evaluation queries |
| Storing full LLM responses in memory | OOM on large evaluation sets | Stream responses to disk, store only extracted metrics in memory | >1000 queries with long responses |
| Running all attack variants x all defense thresholds x all metrics | Combinatorial explosion of experiments | Define primary experiments first; run secondary sweep only on promising configurations | >3 attack types x >5 threshold values |

## Methodology Mistakes

These are specific to academic security research and directly affect your grade.

| Mistake | Risk | Prevention |
|---------|------|------------|
| No threat model definition | Reviewers cannot evaluate whether attacks are realistic | Write a 1-paragraph threat model: who is the attacker, what do they control, what is their goal |
| Cherry-picked examples in presentation | Undermines credibility; instructor will ask about average-case | Always show aggregate metrics first, then examples. Show failure cases too |
| Comparing defense to no baseline | Cannot tell if defense is better than trivial approaches | Include baselines: no defense, keyword filter, random filtering |
| No statistical significance | Single-run ASR is meaningless if it varies by 10% between runs | Run 3+ seeds minimum; report mean and std dev |
| Evaluation on <50 queries | Insufficient statistical power | Use 200+ queries minimum; 500+ preferred |
| Reporting only accuracy, not precision/recall | Hides whether defense catches attacks vs. wrongly blocks clean content | Report precision, recall, F1 for the defense classifier; plot precision-recall curve |

## "Looks Done But Isn't" Checklist

- [ ] **RAG Pipeline:** Often missing chunk overlap configuration -- verify chunking uses overlap (e.g., 200-char overlap on 1000-char chunks) to avoid splitting critical content
- [ ] **Attack injection:** Often missing control over WHERE in the corpus poisoned docs land -- verify you can control and report the poisoning ratio and position
- [ ] **ASR measurement:** Often missing the "what counts as a successful attack" definition -- verify you have a clear, automatable success criterion (exact match? substring? LLM-as-judge?)
- [ ] **Defense classifier:** Often missing the class balance check -- verify training data has roughly balanced positive/negative examples, or use appropriate loss weighting
- [ ] **Evaluation set:** Often missing benign queries that are topically SIMILAR to attack-targeted queries -- verify clean queries cover the same topics as attacked queries
- [ ] **Plots and figures:** Often missing axis labels, legends, or units -- verify every plot has labeled axes and can be understood without reading surrounding text
- [ ] **Presentation:** Often missing a live demo or concrete attack example -- verify you have at least one end-to-end walkthrough (query -> retrieval -> injection -> hijacked output)

## Recovery Strategies

| Pitfall | Recovery Cost | Recovery Steps |
|---------|---------------|----------------|
| Trivially detectable attacks | MEDIUM | Design 2-3 obfuscated attack variants; re-run ASR experiments; 1-2 days of work |
| No retrieval logging | LOW | Add logging to retrieval function; re-run experiments with logging; decompose existing ASR |
| Overfitted defense evaluation | HIGH | Design new held-out attack category; may need to retrain defense; re-run all evaluation; 2-3 days |
| No utility metrics | MEDIUM | Create benign evaluation set from corpus; run defense on clean queries; measure answer quality; 1-2 days |
| Toy corpus | HIGH | Switch to NQ or MS MARCO subset; re-embed; re-run all experiments; 3-4 days |
| No reproducibility | MEDIUM | Retroactively pin seeds and versions; create config file; re-run to verify reproducibility; 1-2 days |
| Conflated threat model | HIGH | Redesign attack payloads to include instruction-override component; re-run everything; 2-3 days |
